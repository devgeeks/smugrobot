Motion

Motion communicates state changes — it is never decorative. All transitions must respect prefers-reduced-motion (durations are set to 0ms automatically via tokens.css). Default easing is --ease-out.

Focus & Shadows

Every interactive element must have a :focus-visible state using box-shadow: var(--focus-ring). Never use outline for focus — it cannot be rounded to match the element. The double-ring ensures visibility on any background.

Design decisions

Why things are the way they are — the reasoning behind choices that aren't obvious from the code alone.

Zero dependencies / native Web Components

Every component extends HTMLElement with Shadow DOM. No framework, no bundler required at runtime. This means the library works in any app — React, Vue, Svelte, plain HTML — without adapter packages or version conflicts. Shadow DOM gives strong style isolation: app CSS cannot accidentally override component internals.

CSS custom properties cross shadow DOM boundaries

CSS custom properties (variables) inherit through shadow roots automatically — this is intentional in the spec. The TOKEN_BRIDGE pattern in each component is therefore minimal: just box-sizing: border-box and :host { display: block }. It does not need to re-declare all tokens, and nothing outside the shadow root can accidentally override component logic via regular CSS selectors.

Semantic tokens over raw ink values

Components never reference --ink-* values directly. They use --surface-* and --text-* semantic tokens. This means the entire theme can switch (dark ↔ light) by changing 9 token values on [data-theme="light"], and every component updates for free without any component-level CSS changes.

Split accent tokens: --cipher vs --cipher-text

--cipher (#2ECC8F) is vivid and only passes WCAG contrast on dark backgrounds, so it's reserved for fills, borders, and glows. --cipher-text switches to a much darker value (#0F5735) in light mode, achieving 8.62:1 contrast (AAA). The same split applies to warn, danger, and info. This prevents the common mistake of using a vivid brand color as readable text on white.

box-shadow for focus, not outline

outline does not follow border-radius in older browsers and cannot be offset from the element surface. box-shadow: var(--focus-ring) is a double ring — 2px same as background color, 4px cipher green — which creates a visible gap between the element and the ring, making it legible on any background color.

vault-select syncs via MutationObserver

Native <option> elements cannot be projected into a shadow DOM <select> via <slot> — browsers do not allow slotted options. The component instead watches its light DOM children via MutationObserver and clones them into the shadow <select> on every change. This lets callers write idiomatic <option> markup while still getting the styled wrapper.

--cipher signals trust only — never decorative

Cipher green is the system's primary signal for "verified", "encrypted", "active". Using it decoratively (e.g. a green header for branding) would erode its meaning. In a privacy-focused product, users need to be able to trust that a green indicator means something specific.

Two font roles, no mixing

Geist Mono for all UI chrome (labels, headings, code, button text, badges). Geist Sans for all human-readable content (prose, input values, descriptions). The contrast between a monospaced label and a proportional body text is intentional — it makes the boundary between "system" and "user content" visually clear.

Events

All custom events bubble out of shadow DOM (composed: true). Listen on the component element or any ancestor.

Event Fired by Detail shape When
vault-input vault-input, vault-textarea { value: string } Every keystroke
vault-change vault-input, vault-textarea, vault-select { value: string } On blur / commit
vault-change vault-toggle { checked: boolean } On toggle
vault-dismiss vault-alert undefined Dismiss button clicked
vault-open vault-popover undefined Popover becomes visible
vault-close vault-popover undefined Popover is dismissed (outside click, Escape, or .close())

Theming

Theme is controlled by data-theme="light" on <html>. Default is dark. The VaultTheme helper persists the preference to localStorage and can follow OS preference. 

import { VaultTheme } from '../src/components/vault.js';

VaultTheme.init();        // call once on boot — restores saved preference
VaultTheme.set('dark');   // force dark
VaultTheme.set('light');  // force light
VaultTheme.set(null);     // follow OS preference (removes localStorage entry)
VaultTheme.get();         // returns 'dark' | 'light'
Why data-theme on <html> and not :root

[data-theme="light"] on the root <html> element keeps the toggle selector simple and lets any part of the page (including shadow roots) inherit the correct semantic token values, since CSS custom properties cascade down the full document tree. It is also easy to override for sub-sections: put data-theme="dark" on any ancestor element to force dark mode in just that region.
